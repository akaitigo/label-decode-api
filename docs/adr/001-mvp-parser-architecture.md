# ADR-001: MVP パーサーアーキテクチャ

## ステータス
承認済み

## コンテキスト
食品表示テキストの構造化パースには自然言語処理が必要。当初の tech_stack では Python NLP パーサーを Kotlin gRPC サーバーから呼び出す設計だったが、MVP では複雑度を抑える必要がある。

## 決定
**MVP ではルールベースのパーサーを Kotlin で直接実装する。** Python NLP パーサーは v2 で導入する。

### 理由
1. Kotlin-Python 間のプロセス呼び出しは運用複雑度が高い（デプロイ、エラーハンドリング、パフォーマンス）
2. 日本の食品表示法は規格化されており、スラッシュルールや括弧表記はルールベースで十分に解析可能
3. MVP の目標は「動く API を出すこと」であり、NLP の精度は v2 の改善対象

### v2 への移行計画
1. `parser/` ディレクトリに Python NLP パーサーを実装
2. Kotlin パーサーと並行稼働し、精度を比較
3. NLP パーサーが優位な場合、gRPC サービスとして分離

## 結果
- MVP のデプロイが Kotlin 単体で完結し、シンプルになる
- parser/ ディレクトリは v2 用に予約
- ルールベースの限界（非標準的な表記、誤字）は v2 で対応
