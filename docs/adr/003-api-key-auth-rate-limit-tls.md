# ADR-003: 本番化セキュリティ（APIキー認証・レート制限・TLS）

## ステータス
承認済み

## コンテキスト
MVP は「認証なし・レート制限なし・平文 gRPC」で出荷した（README に本番利用前の警告を明記）。
本番利用に向けて、この 3 点を解消する必要がある。前提条件:

- デプロイ先は GCP Cloud Run（単一〜少数インスタンス、フルマネージド）
- 外部サービスの従量課金を発生させない（Free tier で完結）
- クライアントはサーバー間連携（B2B API）であり、エンドユーザーの個人アカウントは存在しない

## 決定

### 1. 認証: 静的 API キー（`x-api-key` メタデータ）
gRPC の `@GlobalInterceptor`（`ApiKeyAuthInterceptor`）で全 RPC を保護する。

- キーは `API_KEYS` 環境変数（カンマ区切りで複数可、ローテーション用）で注入。ハードコード禁止
- 認証失敗は `UNAUTHENTICATED`（HTTP 401 相当）
- キー照合は `MessageDigest.isEqual` による定数時間比較（タイミング攻撃対策）
- 認証有効かつキー未設定なら起動失敗（fail-fast）。認証なしで本番起動する事故を防ぐ
- `%dev` / `%test` プロファイルではデフォルト無効（ローカル開発の摩擦を排除）

**代替案と不採用理由**
- OIDC / JWT（Keycloak 等）: IdP の運用が必要でコストと複雑度が過大。B2B の少数クライアントには不要
- mTLS クライアント証明書: 証明書配布・失効管理が重い。Cloud Run のマネージド TLS 終端とも相性が悪い

### 2. レート制限: インメモリ・トークンバケット
`RateLimitInterceptor` + `TokenBucketRateLimiter`（自前実装、約 80 行）。

- クライアント識別: API キー単位（認証無効時は接続元アドレス単位）
- 既定値: 120 リクエスト/分、バースト容量 30（環境変数で調整可）
- 超過は `RESOURCE_EXHAUSTED`（HTTP 429 相当）
- バケット追跡は 10,000 クライアント上限。超過時は回復済みの古いバケットを破棄（メモリ枯渇防止）
- インターセプタ優先度: 認証(100) → レート制限(50)。無効キーのフラッドは認証で先に遮断される

**代替案と不採用理由**
- Redis ベース（分散レート制限）: インスタンス追加の課金が発生。単一インスタンス前提の現段階では不要
- Bucket4j 等のライブラリ: 依存追加に対して要件が小さすぎる（バケット 1 種、分散不要）

**既知の制約**: インメモリのためインスタンス単位の制限になる。Cloud Run で複数インスタンスに
スケールアウトした場合、実効上限は「設定値 × インスタンス数」になる。厳密な全体制限が必要に
なった時点で Redis 等への移行を再検討する（本 ADR の見直しトリガー）。

### 3. TLS: Quarkus 設定による有効化
証明書は環境依存のため、環境変数で PEM 証明書/鍵のパスを注入する方式とする。

```
QUARKUS_GRPC_SERVER_SSL_CERTIFICATE=/etc/tls/server.crt
QUARKUS_GRPC_SERVER_SSL_KEY=/etc/tls/server.key
```

（`quarkus.grpc.server.ssl.certificate` / `.key` と同義。設定すると gRPC サーバーは TLS で待ち受け、
平文接続を受け付けない。）

Cloud Run へのデプロイ時は Google のフロントエンドが TLS を終端するため必須ではないが、
VM・オンプレ等で直接公開する場合は上記の設定で終端する。自己署名証明書での動作検証手順は
README に記載。

## 結果
- README の MVP 警告（認証・レート制限・TLS）を解消
- 外部サービス・追加課金なしで本番相当のセキュリティ基線を確保
- API キーの発行・失効は環境変数の更新 + 再デプロイで運用（キー管理サービスの導入は将来課題）
- テスト: 認証（キーなし/不正/正/複数キー）、レート制限（バースト内/超過）、リミッタ単体（補充・容量上限・クライアント分離・eviction）を追加
